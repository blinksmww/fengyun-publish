## 图注模式

### 是否使用图注

6 篇均无独立的 `<figcaption>` 元素。图注信息通过两种方式承载：

**方式 1：`alt` 属性（主要方式）**

所有正文图片都有 `alt` 文本，内容并非仅仅是文件名，而是对图片内容的实质描述。

典型 alt 文本示例：
- `alt="Artificial Analysis 推文，Qwen3.7-Max 在 Intelligence Index 上拿到 56.6 分"`（篇 A）
- `alt="适应峰示意图：人形从「当前位置」下山，经过「必经的下行」，再爬上「更高的山」"`（篇 B）
- `alt="Claude Code使用分析dashboard：131亿token / 94天活跃 / 1130日均消息 / 94.3% cache命中率，4月用量曲线明显跃升"`（篇 C）
- `alt="从「我要怎么把事做对」到「这个世界到底是怎么运转的」：5269 条划线 7 年主题地图"`（篇 F）

`alt` 文本平均字数约 20–40 字，含上下文信息（数据量级、关键结论、场景描述）。

**方式 2：图片前后的叙述段落**

每张图片前（上文）通常有 1–2 句引导语，图片后通常有 1–2 段解读文字。花叔的风格是：图是「证据」，文字是「判断」，不依赖图注，靠文段呼应。

**结论：6/6 篇无 figcaption 图注，但 alt 属性承担图注角色，且质量极高（详述性 alt，不是装饰性 alt）。**

## 图片尺寸特征

### inline style 汇总

**技术文章（篇 A/C/D/F）图片通用 style：**
```
max-height: 500px !important;
height: auto;
display: block;
margin: 24px auto;
border-radius: 10px;
box-shadow: 0 6px 24px rgba(193, 95, 60, 0.1);
```

**思想文章（篇 B/E）图片通用 style（篇 B 与 A 共用排版）：**
```
display: block;
margin: 24px auto;
border-radius: 10px;
box-shadow: rgba(193, 95, 60, 0.1) 0px 6px 24px;
max-height: 500px !important;
pointer-events: initial;
```

**篇 F（深红配色文章）图片有差异：**
```
max-height: 500px !important;
height: auto;
display: block;
margin: 24px auto;
border-radius: 6px;
border: 2px solid #d32f2f;
box-shadow: 0 4px 12px rgba(211, 47, 47, 0.12);
```
篇 F 的图片加了 `border: 2px solid #d32f2f`（深红色边框），其他 5 篇无 border。border-radius 也从 10px 降为 6px。

### 尺寸控制方式

- 全部使用 `max-width: 100%`（全局 CSS 级别）+ `max-height: 500px`（inline 级别）双重约束。
- `display: block; margin: 24px auto` 意味着所有图片均为块级居中，不存在浮动或左右对齐图。
- 原始图片宽度信息保存在 `data-w` 属性（如 `data-w="1080"`），实际渲染宽度受容器（max-width: 667px）约束，均以全容器宽度展示。
- **无任何图片是小图或局部图**——全部按宽度铺满，没有缩略图、没有列排图、没有 50% 宽度图。

### GIF 特殊处理

篇 A 和篇 C 各有 GIF，其 section 容器有 `text-align: center`，但 GIF 本身无额外 style，沿用全局 `max-width: 100%`。

## 关键发现

1. **平均每 6–7 段 1 张图**（范围 4.5–8.8）；技术演示文章偏密（4–5 段/图），思想论证文章偏稀（8–9 段/图）。

2. **首图位置惯例：正文最多 3–5 段内必出第 1 张图**，没有一篇超过 300 字的导言后才出图。篇 F 甚至是第 2 行即出图。首图永远是「话题的实物/数据凭证」，不是装饰图。

3. **图注使用率：0/6 篇有 figcaption**，但 6/6 篇 alt 文本承担了详述型图注功能。alt 文本相当于「给看不见图的读者」写的一句话摘要，含关键数据或结论，非装饰性填词。

4. **图片类型主力：产品截图占 50%，AI 生成插画占 29%**。截图几乎只出现在「有产品可演示」的文章；纯思想文章（篇 E）100% 用 AI 生成插画替代截图。两种类型的使用场景完全不重叠，呈功能性分工。

5. **末图不一定是收尾图**：6 篇中只有篇 A 末图在文章最后一个 H2 末（有一定收尾感），其余 4–5 篇末图后都还有 3–10 段文字。花叔不依赖「末图收尾」的惯例。

6. **无一张装饰图、无一张纯美观插图（除 AI 插画外）**：所有截图都是实际操作截图，所有 AI 插画都对应正文中的一个概念（适应峰/红皇后/寒武纪/中世纪金匠/科斯），不存在与内容无关的氛围图。

7. **尺寸铁律：全宽块级居中 + max-height 500px**，6 篇完全统一。例外仅 1 处：篇 F 图片有 2px 深红色 border，与该篇整体配色系统一致（篇 F 是唯一深红配色文章）。

## 不确定的地方

1. **字数估算误差**：正文字数为从 HTML 段落内容人工估算，未精确提取文本节点字数，误差约 ±15%，字图比数据仅供参考量级。

2. **篇 A 图片数存疑**：assets 目录有 10 张 PNG + 3 张 JPG + 2 张 GIF = 15 个媒体文件，但其中 cover_image（1 张 JPG）和 novel-card 占位图（2 张内嵌 `<img>`）不计入正文图。此外有 2 张 PNG（17796124764520 和 17796124765820）在 HTML 中未见明确对应的 `<img src>` 引用，可能是微信内置备用资源或已删除段落的残留，本报告未将这 2 张计入。

3. **篇 B 首图 inline 性质**：篇 B 首张图是 `<img>` 直接嵌在 `<p>` 标签内（非独立 section/块），与其他篇的块级图不同。这是花叔故意为之（引入截图为写作背景），还是排版操作差异，无法从 HTML 单独判断。

4. **AI 插画是花叔自制还是第三方**：篇 E 的 6 张插画风格一致（历史感、cream 底色、信息图鉴风），疑为同一工具批量生成，但来源无法从 HTML 确认（`data-src` 均为 mmbiz.qpic.cn CDN 链接，不含生成工具信息）。
