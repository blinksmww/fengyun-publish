## 用户真实负载评估(用作配置选型依据)

### 常驻服务
- we-mp-rss Docker: 300-500MB
- RSSHub Docker: 500-800MB
- TrendRadar: 200MB
- 系统 + Docker daemon: 400MB
- **常驻总计: 1.5-2GB**

### cron 峰值(每天 1 篇 fengyun-publish)
- Claude Code 9 步流程: 1.5-2GB
- **峰值总计: 3.5-4GB**

### 网络
- cron 拉 60 个 RSS ≈ 20-50MB/次
- Claude API 调用: 1-2MB/次
- **月总流量 < 5GB**, 远低于任何套餐限制

### CPU
- I/O bound 为主, 2 核够用

### 结论
- **2C4G 边界够用**, 4G swap 兜底
- **4C8G 是过度配置**, 浪费

## Musk × Jobs 辩论

### Musk · 物理 + Idiot Index

- **4C8G 多花 ¥260+ 不带来 fengyun-publish 质变** → Idiot Index 高, 不选
- 性能上 2C4G 全部厂商够用, 真正问题是「同档哪家便宜 + 稳定 + 续费同价 + 报销好开发票」
- 火山引擎 ¥89 物理最划算, 但**字节系活动结束后涨价历史比阿里腾讯凶**, 长期风险高
- 腾讯云 ¥188 + 3 个月 = 等效 ¥150/年, 比阿里 ¥199 省 ¥50
- 阿里 200M 不限流量对用户场景**无实际价值**(用不上)

### Jobs · 产品 + 生态 + 长期

- **微信公众号 IP = 腾讯系生态**, 腾讯云原话「无缝对接微信小程序」
- 阿里云**稳定性 + 文档 + 工单响应**国内最佳, **是 Day 4-5 部署遇到坑时的救命依赖**
- 报销维度: 阿里 / 腾讯发票兼容最好, **火山引擎对学校报销可能有兼容性问题**
- 长期看 ¥50/年 不值得纠结, 选「遇到问题最容易找到答案的那家」
